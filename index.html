Режимы и команды

3 концепции
режим +
Plan mode
Claude сначала думает и составляет план — потом делает.

Перед тем как Claude начнёт писать код, он составляет подробный план. Ты видишь шаги заранее, можешь их поправить, и только потом Claude идёт в работу.

Ключевой трюк: дай Claude способ проверить результат. Тест, команда, URL. Без этого он будет угадывать «готово».

Think step by step. Write a plan first, then implement. Include how to verify each step. Wait for my approval.
паттерн +
/loop — итеративные задачи
Повторяющиеся задачи: файл за файлом, тест за тестом.

Важно: /loop — не встроенная команда. Это паттерн, который ты описываешь в промпте или CLAUDE.md.

Суть: Claude делает одну единицу работы → проверяет → повторяет. Останавливается когда тест упал или задача завершена.

Work through this iteratively. For each file: 1. Make the change 2. Run: npm test 3. If passes — next file 4. If fails — STOP and report Never continue after a failure.
команда +
/btw — побочный вопрос
Задать вопрос пока Claude работает, не прерывая основную задачу.

Пока Claude реализует что-то большое, можно «в сторону» спросить уточняющий вопрос. Claude ответит и продолжит работу.

Это как сказать коллеге «кстати...» не останавливая его от основной работы.

/btw What naming convention should I use for these utils?

Файлы и структура

3 инструмента
git +
Git Worktrees
Параллельная работа в нескольких ветках одновременно.

Один репозиторий — несколько папок на разных ветках. Запусти Claude Code в двух окнах: один над фичей A, второй над фичей B.

# Создать новый worktree git worktree add ../myapp-feature-b feature-b # Открой в Claude Code cd ../myapp-feature-b
практика +
Этапы + гейты
Тест-чекпоинты между этапами. Не иди дальше пока не прошёл.

Разбей задачу на фазы и добавь обязательную проверку (гейт) после каждой. Claude не переходит к следующей фазе пока гейт не пройден.

Phase 1: Create database schema Gate: run npm run db:validate — must pass Phase 2: Build API endpoints Gate: run npm test api — all green Phase 3: Connect frontend Gate: manually test login flow
ревью +
Code Review в новом контексте
Свежий чат находит баги которые пропустил исходный агент.

Когда Claude долго работал над кодом — он «замыливается». Открой новый чат, вставь код и попроси ревью. Свежий контекст = другой взгляд.

Review this code for bugs and edge cases. You have no context about how it was written. Be critical. Format: CRITICAL / IMPORTANT / MINOR

CLAUDE.md

Инструкция для Claude по проекту
Лимит: 200 строк на файл. Больше — Claude начинает игнорировать содержимое. Пиши по факту проблем, не заранее. Claude читает CLAUDE.md из текущей папки и всех родительских.
Project обязательно
Одна строка: что за проект, стек. Claude читает это первым.
Setup обязательно
Как установить зависимости и запустить проект.
Testing обязательно
Команда для запуска тестов. Без этого Claude не проверит работу.
Architecture обязательно
Где что лежит. API, компоненты, конфиги.
Code style полезно
Только то что отличается от общепринятого. TypeScript strict? no any?
Do not полезно
Явные запреты с альтернативами. «Не X — используй Y».
Commands по желанию
Пользовательские команды: /loop, /review и т.д.
When stuck полезно
Когда остановиться и спросить. Предотвращает тихий рефакторинг.
Минимальный шаблон
~30 строк — достаточно для большинства проектов
+
# [Название проекта] ## Project Next.js 14 + TypeScript. E-commerce platform. ## Setup npm install npm run dev # localhost:3000 ## Testing npm test # run after every change npm run typecheck # npx tsc --noEmit ## Architecture /src/app # Next.js pages & routes /src/components # React components /src/lib # utilities, helpers /src/api # API handlers only ## Rules - TypeScript strict, no any - API calls only in /src/api - Run npm test before marking done
Полный шаблон с командами
~120 строк — для больших проектов
+
# MyApp — CLAUDE.md ## Project Node.js + React SaaS. TypeScript throughout. B2B task management. Multi-tenant, Postgres. ## Setup npm install cp .env.example .env npm run db:migrate npm run dev # API :4000, UI :3000 ## Testing npm test # jest, run after changes npm run typecheck # must pass before done ## Architecture /apps/api/src/routes # Express routers /apps/api/src/services # business logic /apps/api/src/db # Prisma queries only /apps/web/src/features # feature modules /apps/web/src/api # API client calls ## Code style - TypeScript strict, no any - Services return Result<T, Error>, never throw - CSS: Tailwind only, no inline styles ## Do not - Modify /prisma/schema.prisma without asking - Use console.log — use /lib/logger instead - Change auth logic in /middleware/auth.ts ## When stuck - Ask before architectural decisions - Stop if tests fail after 2 attempts - Show plan if change touches >3 files ## Commands ### /review Check last change for TypeScript errors, missing error handling, security issues. Report as CRITICAL / IMPORTANT / MINOR. ### /loop [task] Break into units, confirm list, process one at a time with npm test after each. Stop if tests fail. Report every 5 units.
Почему Claude игнорирует CLAUDE.md
5 частых причин и как исправить
+
Слишком длинный
Больше 200 строк — Claude перестаёт читать внимательно. Разбей на несколько файлов по папкам.
Правила без альтернативы
«Не используй console.log» — игнорирует. «Не используй console.log — используй /lib/logger» — следует.
Нет команды тестирования
Без явного «run npm test after changes» Claude не проверяет работу самостоятельно.
Абстрактная архитектура
«API в отдельном слое» непонятно. «API calls only in /src/api, never in components» — однозначно.
Написан заранее «про запас»
Добавляй правило только когда Claude реально сделал что-то не так. Не фантазируй заранее.

Агенты и суб-агенты

Command vs Agent vs Skill
команды +
Command — предсказуемый workflow
Для повторяющихся задач с известными шагами. Определяешь один раз, запускаешь командой.

Используй когда задача всегда делается одинаково: /review, /loop, /deploy. Лучше чем суб-агент для предсказуемых задач — меньше токенов, точнее результат.

## Commands ### /review 1. Check TypeScript correctness 2. Find missing error handling 3. Flag security issues Report as CRITICAL / IMPORTANT / MINOR
агенты +
Узкие суб-агенты
Специалист лучше универсала. Один агент — одна конкретная задача.

Вместо «QA агент» (слишком широко) создай конкретного специалиста. Чем уже задача — тем точнее результат.

# Плохо — слишком широко /agent qa-engineer "Do quality assurance" # Хорошо — конкретная задача /agent type-checker "Check TypeScript strict mode violations only" /agent sql-reviewer "Find N+1 queries and missing indexes"
AskUser +
AskUserQuestion — интервью
Попроси Claude сначала взять у тебя интервью, потом писать код.

5 минут на вопросы экономят часы переделок. Claude уточняет требования пока не поймёт задачу полностью — только потом показывает спецификацию и начинает работу.

Before starting, use AskUserQuestion to interview me about requirements. Ask until you have enough for a complete spec. Show me the spec before writing any code.
принцип +
Когда просто Claude Code лучше
Для небольших задач — обычный режим лучше сложных workflow.

Все эти workflow и агенты нужны только когда задача реально большая и сложная. Для маленьких изменений они только мешают — больше токенов, дольше, менее точно.

Правило: начни с простого. Добавляй сложность только когда столкнулся с реальной проблемой.

Шаблоны промптов

Plan mode — 3 сценария
Универсальный Plan mode
Подходит для любой задачи
+
## Task [Опиши задачу одним абзацем] ## Plan first, then implement Before writing any code: 1. Analyse the existing codebase 2. Write a numbered plan with all steps 3. For each step, specify how to verify it 4. Wait for my approval before starting ## Verification method After each step, run: [команда] Step is done only when this passes. ## Constraints - Do not modify files outside [папка] - Follow existing code style - Ask if anything is unclear before starting
Почему это работает
«Wait for my approval» — без этого Claude сразу пойдёт писать код. «Verification method» — конкретная команда которую Claude запустит. Без неё он будет угадывать «готово».
Новая фича с фазами
Гейты между фазами — не иди дальше пока тест не прошёл
+
## Feature Add user authentication with email + password. Store sessions in Redis. Protect /dashboard route. ## Plan mode Think through this completely before any code. Write a plan in this format: Phase 1: [название] - Steps: ... - Gate: `npm test auth` must pass before Phase 2 Phase 2: [название] - Steps: ... - Gate: manually test login at localhost:3000/login Phase 3: [название] - Steps: ... - Gate: `npm run test:e2e` all green ## Show me the plan first. I will approve before you start Phase 1.
Баг-фикс
Диагноз перед лечением — минимальное исправление
+
## Bug Users get logged out after page refresh. Happens only in Safari. Works fine in Chrome. ## Reproduce 1. Log in at /login 2. Refresh the page 3. Expected: stay logged in 4. Actual: redirected to /login ## Plan before fixing 1. Investigate — read relevant auth code 2. Form a hypothesis about root cause 3. Write diagnosis as a short summary 4. Propose the minimal fix (no refactoring) 5. Tell me the fix before applying it ## Verification Fix is complete when: - `npm test` passes - Manual test in Safari: login → refresh → logged in
Loop — итеративная обработка файлов
Например: конвертировать все .js в TypeScript
+
## Task Convert all JavaScript files in /src/components to TypeScript. ## Loop instructions Work through this iteratively: 1. List all .js files in /src/components 2. For each file, one at a time: a. Convert to TypeScript (.ts/.tsx) b. Run: npx tsc --noEmit c. If passes — move to next file d. If fails — STOP and report error 3. After every 5 files, summarise: "Done: X/Y files. Current: [filename]" ## Stop conditions - TypeScript error you can't fix in 2 attempts - File requiring architectural decisions - Any test suite failure ## Do not - Change logic, only add types - Skip files or mark them "todo" - Continue after a stop condition

Честно: готового инструмента нет. Это ручной паттерн — Claude пишет, другая модель проверяет независимо. Ты — связующее звено.

Промпт для code review
Вставить в ChatGPT / GPT-4o
+
You are a senior code reviewer. Review the following code change critically. You have no context about how it was written. Focus on: 1. Bugs — logic errors, null/undefined cases 2. Edge cases — what inputs would break this? 3. Security — injection, auth bypass, data exposure 4. Type safety — missing checks, unsafe casts 5. Performance — N+1 queries, unnecessary renders 6. Error handling — what happens when things fail? Format your response as: CRITICAL: [will cause bugs or security problems] IMPORTANT: [fix before merge] MINOR: [style/improvement suggestions] LOOKS GOOD: [what is done well] Do not praise the code unless it deserves it. --- CODE START --- [вставь diff или код файла] --- CODE END ---
Промпт для ревью плана
Поймать проблему до написания кода
+
You are a senior architect. Review this implementation plan critically. Point out what could go wrong before any code. Look for: 1. Missing steps — what did the author forget? 2. Wrong order — steps that should be earlier 3. Hidden dependencies — things that must exist first 4. Scalability — will this break at 10x load? 5. Rollback — can this be safely undone? 6. Testing gaps — what's hard to test? Be specific. Vague concerns are not useful. Format: BLOCKERS: [must fix before starting] RISKS: [watch carefully during implementation] SUGGESTIONS: [optional improvements] SOLID: [what's well thought out] --- PLAN START --- [вставь план от Claude Code] --- PLAN END ---
Скрипт-мост (Python)
Автоматически берёт git diff и отправляет в OpenAI API
+
#!/usr/bin/env python3 # Сохрани как cross_review.py # pip install openai # Нужен OPENAI_API_KEY в env import subprocess, openai, sys def get_diff(): result = subprocess.run( ["git", "diff", "HEAD"], capture_output=True, text=True ) return result.stdout def review(diff: str) -> str: client = openai.OpenAI() response = client.chat.completions.create( model="gpt-4o", messages=[{ "role": "user", "content": f"""You are a senior code reviewer. Review this git diff for bugs, security issues, and edge cases. Be specific and concise. Format: CRITICAL / IMPORTANT / MINOR / LOOKS GOOD {diff}""" }] ) return response.choices[0].message.content if __name__ == "__main__": diff = get_diff() if not diff.strip(): print("No changes to review.") sys.exit(0) print("=== Cross-model review ===") print(review(diff))
Запуск: python cross_review.py после того как Claude Code сделал изменения. Можно добавить как команду /xreview в CLAUDE.md.

Community workflows

Популярные паттерны
workflow +
RPI — Research Plan Implement
Трёхфазный процесс для сложных фич.

Research — Claude изучает кодовую базу и задаёт вопросы.

Plan — составляет детальный план с гейтами.

Implement — только после твоего одобрения плана.

Хорошо для сложных фич где нельзя прыгать сразу в код. Живёт в репозитории shanraisshan/claude-code-best-practice.

workflow +
Ralph Wiggum Loop
Полностью автономная работа над большими задачами.

Петля без участия человека: делает шаг → проверяет результат → корректирует → повторяет.

Полезно для длинных монотонных задач: миграции, рефакторинг по всей базе кода, массовые обновления. Требует очень чётких stop conditions.

tools +
Github Speckit (74K ★)
Из спецификаций — код. И обратно.

Инструмент для работы со спецификациями. Описываешь что должна делать система — Claude генерирует код. Можно взять существующий код, превратить в спецификации и пересоздать из них — полезно при больших рефакторингах.

tools +
obra/superpowers (72K ★)
Расширения и суперспособности для Claude Code.

Набор расширений которые добавляют новые возможности поверх стандартного Claude Code. Много community-contributed инструментов для специфических сценариев.

MCP +
MCP — дополнительные источники
Claude видит логи Chrome, базы данных, внешние сервисы.

MCP (Model Context Protocol) подключает внешние источники данных. Например, MCP для Chrome DevTools — Claude видит console.log и ошибки без копирования.

# ~/.claude/config.json { "mcpServers": { "chrome": { "command": "npx", "args": ["@modelcontextprotocol/server-puppeteer"] } } }
отладка +
Скриншоты при зависании
Если застрял — скриншот вместо описания.

Если что-то не работает и трудно объяснить словами — сделай скриншот и перетащи в чат. Claude Code умеет работать с изображениями. Особенно полезно для CSS-проблем и ошибок в браузере.

Ключевые вопросы

На миллиард долларов
?
Что должно быть внутри CLAUDE.md?
Project (стек), Setup (команды), Testing (как проверить), Architecture (где что лежит), Do not (явные запреты с альтернативами). Максимум 200 строк. Только то что реально нужно.
?
Когда использовать command vs agent vs skill?
Command — для предсказуемых повторяющихся задач (/review, /loop). Agent — для узкоспециализированных задач с гибкостью. Skill — переиспользуемый блок инструкций. Для маленьких задач — просто обычный Claude Code.
?
Почему Claude игнорирует CLAUDE.md?
Файл слишком длинный, правила размытые, нет команды тестирования, нет альтернатив для запретов, или написан заранее «про запас» вместо по факту реальных проблем.
?
Можно ли превратить кодовую базу в спецификации и пересоздать код из них?
Да, это идея Github Speckit и OpenSpec OPSX. Полезно при больших рефакторингах или миграции на новый стек. Но требует качественных спецификаций — мусор на входе, мусор на выходе.
